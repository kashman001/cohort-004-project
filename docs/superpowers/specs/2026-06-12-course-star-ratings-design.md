# Course Star Ratings ‚Äî Design Spec

**Date:** 2026-06-12
**Status:** Approved design, pending implementation plan
**Visual companion:** [`docs/course-star-ratings-design.html`](../../course-star-ratings-design.html)

## Summary

Let enrolled students rate a course from 1 to 5 stars (no written reviews). Show the
average rating ‚Äî a star row, a one-decimal number, and the count ‚Äî everywhere courses
appear, starting with the catalog list page and the course detail page.

Averages are computed live with SQL aggregates (no denormalized columns), matching the
existing service-layer style and the modest local SQLite dataset.

## Decisions (resolved)

| Question | Decision |
|---|---|
| Who can rate | **Enrolled students only** |
| Edit behaviour | **One rating per user per course, editable** (re-submit updates) |
| Input location | **Course detail page** (`courses.$slug.tsx`) |
| Display | **Stars + numeric average + count**, e.g. `‚òÖ‚òÖ‚òÖ‚òÖ‚òÜ 4.2 (37)` |
| Compute strategy | **Live SQL aggregate** (`AVG`/`COUNT`), no cached columns |
| Scale | 1‚Äì5 whole stars (no half-star input) |
| Display rounding | Average to one decimal; star row fills to nearest half |
| Zero ratings | Show "No ratings yet", `average` is `null` (never `0.0`) |
| Input visibility | Interactive stars only for enrolled students; everyone else sees read-only average |

## Scope

**In scope**
- New `course_ratings` table + migration
- New `ratingService.ts` (submit / read / aggregate)
- Reusable component: `StarRating` (read-only) + `StarRatingInput` (interactive)
- Wiring into the catalog list page and the course detail page

**Out of scope (intentionally)**
- Written/text reviews
- Dashboard and home-page placement (the component drops in trivially later)
- Denormalized cached average columns (premature at this scale)

## 1. Data model

One new table in `app/db/schema.ts`, following existing conventions (auto-increment
integer PK, FK `references()`, ISO-string timestamps via `$defaultFn`):

```ts
export const courseRatings = sqliteTable("course_ratings", {
  id: integer("id").primaryKey({ autoIncrement: true }),
  userId: integer("user_id").notNull().references(() => users.id),
  courseId: integer("course_id").notNull().references(() => courses.id),
  stars: integer("stars").notNull(), // 1‚Äì5
  createdAt: text("created_at").notNull().$defaultFn(() => new Date().toISOString()),
  updatedAt: text("updated_at").notNull().$defaultFn(() => new Date().toISOString()),
});
```

A **unique index on `(user_id, course_id)`** enforces "one editable rating per user per
course" and makes submission idempotent via upsert. Migration generated by
`drizzle-kit generate` ‚Üí `drizzle/0003_*.sql`, applied with `drizzle-kit migrate`.

## 2. Service layer ‚Äî `app/services/ratingService.ts`

New file, mirroring the existing style (positional params, direct `db` calls, `sql`
aggregates as in `getEnrollmentCountForCourse`). Routes never touch the table directly.

| Function | Returns | Notes |
|---|---|---|
| `submitRating(userId, courseId, stars)` | saved row | Validates `stars ‚àà 1..5`; verifies enrollment (throws otherwise); upserts on the unique index. |
| `getUserRating(userId, courseId)` | `number \| null` | The caller's current star value, to pre-select the input. |
| `getCourseRatingSummary(courseId)` | `{ average: number \| null, count: number }` | `AVG(stars)`, `COUNT(*)`; `average` is `null` when count is 0. |
| `getRatingSummariesForCourses(courseIds: number[])` | `Map<number, { average, count }>` | One `GROUP BY course_id` query so the list page stays a single round-trip (no N+1). |

Core upsert:

```ts
db.insert(courseRatings)
  .values({ userId, courseId, stars })
  .onConflictDoUpdate({
    target: [courseRatings.userId, courseRatings.courseId],
    set: { stars, updatedAt: new Date().toISOString() },
  })
  .run();
```

## 3. UI component ‚Äî `app/components/star-rating.tsx`

One file exporting two pieces. Stars use the Lucide `Star` icon (already a dependency).

- **`StarRating`** (read-only): given `average` and `count`, renders the star row (filled
  to nearest half), the numeric average to one decimal, and `(count)`. When `count === 0`,
  renders "No ratings yet". Used on both pages.
- **`StarRatingInput`** (interactive): five clickable stars inside a React Router
  `<Form method="post">`, submitting `stars` to the detail route's `action`. Hover preview,
  current value pre-selected. Rendered only for enrolled students.

## 4. Route wiring

**`courses.tsx` (catalog list)**
- Loader: after `buildCourseQuery`, call `getRatingSummariesForCourses(courseIds)` once;
  attach `{ average, count }` to each course.
- Render read-only `<StarRating>` in the `CardFooter`.

**`courses.$slug.tsx` (detail)**
- Loader: add `getCourseRatingSummary(course.id)`; if enrolled, also
  `getUserRating(currentUserId, course.id)`.
- New `action`: read `stars` from form data, call `submitRating(...)`, return
  success/validation result (surfaced via the existing `sonner` toast pattern already in
  the file). React Router revalidation re-runs the loader so the average and the user's
  selection refresh without a manual refetch.
- Render the read-only summary near the title; render `StarRatingInput` in the
  enrolled-student section.

## 5. Edge cases & error handling

| Situation | Behaviour |
|---|---|
| Logged-out / not enrolled | No input rendered; read-only summary still shown. `submitRating` re-checks enrollment server-side (defense in depth). |
| Stars outside 1‚Äì5 | Service throws ‚Üí action returns validation error ‚Üí toast. No row written. |
| Zero ratings | "No ratings yet" everywhere; `average` is `null`, never `0.0`. |
| Re-submitting | Upsert updates the existing row in place ‚Äî never a duplicate. |
| Instructor viewing own course | Not enrolled ‚Üí no input, by the same rule. No special-casing. |

## 6. Testing

Following the existing `*.test.ts` service-test pattern (e.g. `enrollmentService.test.ts`).
The logic lives in the service, so the tests do too; the UI is thin glue.

`app/services/ratingService.test.ts`:
- `submitRating` creates a rating for an enrolled user
- Re-submitting updates the value (asserts one row, not two)
- A non-enrolled user is rejected
- Out-of-range `stars` rejected
- Average & count math correct, including the zero-rating `null` case
- `getRatingSummariesForCourses` groups multiple courses correctly in one query
