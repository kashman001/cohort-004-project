Cadence Course Platform · Feature Design

Course Star Ratings

Enrolled students rate a course from 1 to 5 stars. The average rating — stars, a one-decimal number, and the count — appears everywhere courses are shown, starting with the catalog list and the course detail page.

Who rates · Enrolled students Rating · 1 per user, editable Input · Course detail page Display · ★ + number + count Compute · SQL aggregate (live)

01Overview & scope

A single new table, one new service, and one reusable component. No written reviews — just a star value. Averages are computed live with SQL aggregates (no denormalized columns to keep in sync), which fits the existing service-layer style and the modest local SQLite dataset.

High-level shape: routes call the new ratingService; the service is the only code that touches the course_ratings table; both pages render the shared StarRating component.

In scope

  • New course_ratings table + migration
  • New ratingService.ts (submit / read / aggregate)
  • Reusable StarRating (read-only) and StarRatingInput (interactive) component
  • Wiring into the catalog list and course detail pages

Out of scope (intentionally)

  • Written/text reviews
  • Dashboard & home-page placement — the component drops in trivially later
  • Denormalized cached average columns (premature at this scale)

02Data model

One table, following existing schema conventions (auto-increment integer PK, FK references(), ISO-string timestamps via $defaultFn). "One editable rating per user per course" is enforced by a unique index on (user_id, course_id) and upsert logic in the service.

A join table between users and courses. The unique constraint is the heart of "one editable rating per user."

// app/db/schema.ts — appended
export const courseRatings = sqliteTable("course_ratings", {
  id: integer("id").primaryKey({ autoIncrement: true }),
  userId: integer("user_id").notNull().references(() => users.id),
  courseId: integer("course_id").notNull().references(() => courses.id),
  stars: integer("stars").notNull(),                // 1–5
  createdAt: text("created_at").notNull().$defaultFn(() => new Date().toISOString()),
  updatedAt: text("updated_at").notNull().$defaultFn(() => new Date().toISOString()),
});

03Service layer

A new app/services/ratingService.ts mirrors the existing style — positional params, direct db calls, sql aggregates (just like getEnrollmentCountForCourse). Routes never touch the table directly.

FunctionReturnsNotes
submitRating(userId, courseId, stars) the saved row Validates stars ∈ 1..5; verifies enrollment (throws otherwise); upserts on the unique index.
getUserRating(userId, courseId) number | null The caller's current star value, to pre-select the input.
getCourseRatingSummary(courseId) { average, count } AVG(stars), COUNT(*). average is null when count is 0.
getRatingSummariesForCourses(ids[]) Map<id, {average,count}> One GROUP BY course_id query → list page stays a single round-trip (no N+1). 
// the upsert at the core of submitRating()
db.insert(courseRatings)
  .values({ userId, courseId, stars })
  .onConflictDoUpdate({
    target: [courseRatings.userId, courseRatings.courseId],
    set: { stars, updatedAt: new Date().toISOString() },
  })
  .run();

04Submit flow

An enrolled student clicks a star on the detail page. The interactive component is a React Router <Form method="post"> that hits the route's action — no separate API route needed.

The same loader that rendered the page re-runs after the action (React Router revalidation), so the average and the user's selection update without a manual refetch.

05Read / display flow

Reads differ by page. The list page batches; the detail page fetches one summary plus the viewer's own rating.

06UI component

One file, app/components/star-rating.tsx, exporting two pieces. Stars use the Lucide Star icon already in the dependency set. The read-only view fills to the nearest half star; the average shows to one decimal.

Both modes share the same star-drawing primitive. Logged-out / non-enrolled viewers see only the read-only summary — the input simply isn't rendered (and the service re-checks enrollment server-side).

07Edge cases & error handling

SituationBehaviour
Logged-out / not enrolledNo input rendered; read-only summary still shown. submitRating re-checks enrollment server-side (defense in depth).
Stars outside 1–5Service throws → action returns a validation error → sonner toast. No row written.
Zero ratings"No ratings yet" everywhere; average is null, never 0.0.
Re-submittingUpsert updates the existing row in place — never a duplicate.
Instructor viewing own courseNot enrolled → no input, by the same rule. No special-casing.

08Testing

Following the existing *.test.ts service-test pattern (e.g. enrollmentService.test.ts). The service holds the logic, so it holds the tests; the UI is thin glue.

  • submitRating creates a rating for an enrolled user
  • Re-submitting updates the value (asserts one row, not two)
  • A non-enrolled user is rejected
  • Out-of-range stars rejected
  • Average & count math is correct — including the zero-rating null case
  • getRatingSummariesForCourses groups multiple courses correctly in one query
